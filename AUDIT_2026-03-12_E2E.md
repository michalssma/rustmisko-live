## 2. TECHNICKÝ AUDIT — NALEZENÉ BUGY A RIZIKA

### 🔴 KRITICKÉ (způsobují přímé ztráty)

#### BUG T1: Generic `esports::` edge prochází přes promotion gate
**Stav:** AKTIVNÍ — generuje 73% všech betů a 44% všech ztrát

Přestože `BLOCK_GENERIC_ESPORTS_BETS = true`, bety procházejí pokud:
- `esports_confidence == "high"` AND `esports_reason == "live_payload_sport"`
- Feed-hub propaguje `sport: "cs2"` z trusted sources (dust2, hltv) ale match_key zůstává `esports::`
- Model pak používá CS2 pravděpodobnostní tabulky na zápas s neověřenou sport identitou

**Impact:** 37W/56L, -$15.46 — model nemá spolehlivá data pro tyto zápasy.
System sází na `esports::eatersrune_vs_hashiras`, `esports::quazar_vs_rustecmpkbk` atd. — match_keys kde ani NEVÍME jaký je sport s jistotou.

**Fix:** Viz Sekce 4 - Akční plán.

#### BUG T2: CS2 explicitní path (71.4% WR) nedostává traffic
**Stav:** AKTIVNÍ — 0 betů cs2:: od 10. března

Match_key zůstává `esports::` i pro CS2 zápasy díky normalizaci ve feed-hub. Esports score-edge pak projde přes promotion gate, ale nikdy neprojde jako `cs2::`.

**Root cause:** Feed-hub `match_key()` konvertuje sport → `"esports"` pro všechny esportové sporty při konstrukci klíče. Následná promotion v alert_bot změní `sport` field, ale NE match_key.

#### BUG T3: Esports edge sází underdog odds ≥2.0 (34.5% WR)
**Stav:** AKTIVNÍ — 55 betů v této zóně, zhruba polovina celkového esports volume

Model nadhodnocuje edge u underdogů. Pravděpodobnostní tabulky (round-diff → win probability) nebere v úvahu:
- Azuro vig/margin
- Neověřenou kvalitu score dat
- Fakt že high-odds má implicitně nižší reliabilitu

#### BUG T4: Team name substring matching — false positives
**Lokace:** `team_name_matches_single()` (line ~2138)

```rust
if a.contains(&b) || b.contains(&a) { return true; }
```

Příklady nebezpečných false-match:
- `"navi"` matches `"navegadores"` (substring)
- `"g2"` matches `"g2esports"` (OK) ale i `"tag2"` (NE-OK)
- `"leo"` matches `"leonidas"` (substring)

Pokud oba týmy mají substring overlap s různými Azuro stranami → ambiguita je detekována a BLOCK.
Pokud JEN JEDEN tým matchne → systém sází na potenciálně ŠPATNÝ tým.

**Guard:** `resolve_azuro_side()` vyžaduje unambiguous match (m1 XOR m2). Ale substring false-positive prochází.

#### BUG T5: Stale anomaly outcome_id
**Lokace:** line ~7576

Outcome_id je zafixován v momentě detekce anomálie. Pokud uplyne 10-30 sekund:
- Score se změní
- Vedoucí tým se může převrátit
- Systém sází na staré outcome_id (originální favorit, nyní prohrávající)

**Guard:** Partial — freshness check existuje, ale 30s window je moc velký.

### 🟠 VYSOKÉ RIZIKO

#### BUG T6: align_teams() esports loophole — forced score
**Lokace:** line ~4665

```rust
if is_esports && normal_score == 1 { normal_score = 2; }
```

Pokud z dvou týmů matchne jen jeden, system PŘEDPOKLÁDÁ že je to ten správný zápas. To může projít i pro jiný zápas kde má jeden tým stejné jméno.

#### BUG T7: WS cache stale market version
**Lokace:** line ~7620

Condition freshness check nevaliduje VERZI marketu. Pokud se map_winner condition změní (nový map), WS cache může držet starý condition_id.

#### BUG T8: Remap execution IDs — žádný freshness ordering
**Lokace:** line ~5190

Při retry remappingu systém bere PRVNÍ matching condition z state, ne NEJNOVĚJŠÍ. Může vrátit starý/neaktivní condition.

### 🟡 STŘEDNÍ RIZIKO

#### BUG T9: Tennis anomaly má matematický edge ale P&L záporný
**Příčina:** Avg win odds ~1.58 → breakeven ~63.3% → actual 66.7% ✅
Záporný P&L (-$2.94) je z early období s vyššími stakes a horší WR. Posledních 10 betů: 7W/3L.

**Toto NENÍ bug — je to optimizační příležitost.** Tennis anomaly má REÁLNÝ edge ~3.4%.

#### BUG T10: Football edge model — minute parsing
**Stav:** Football edge je vypnutý. Pokud se znovu zapne, guard musí validovat `detailed_score` formát.

## 4. AKČNÍ PLÁN — CO UDĚLAT TEĎKA

### ⚡ OKAMŽITĚ (Dnes)

#### A1: ZABÍT esports:: edge pro odds ≥ 2.0
**Priorita: KRITICKÁ**
Esports edge underdog bety mají 34.5% WR — každý takový bet je ztrátový.

**Fix:** V `find_score_edges()`, pro `esports::` match_keys přidat hard cap `max_odds = 1.95`.

#### A2: ZPŘÍSNIT esports promotion gate
**Priorita: KRITICKÁ**
Momentálně `"live_payload_sport"` confidence stačí. Ale feed-hub sport propagace je nespolehlivá.

**Fix:** Vyžadovat `esports_confidence == "high"` AND (`esports_reason == "match_key_prefix"` ONLY). To znamená: BET JENOM na `cs2::`, `dota-2::` explicit match_keys, NIKDY na promoted `esports::`.

#### A3: ZVÝŠIT tennis anomaly stakes
**Priorita: VYSOKÁ**
66.7% WR na avg 1.58 odds = +3.4% edge. Ale stakes $0.5 generují jen ~$0.10 profit per win.

**Fix:** Tennis anomaly = primary earner → zvýšit budget allocation na 50% (z aktuálních ~30%).

### 📋 TENTO TÝDEN

#### B1: Opravit feed-hub match_key pro explicitní sporty
Pokud feed-hub VÍ že sport = cs2, match_key by měl být `cs2::` ne `esports::`.
To by odemklo CS2 explicit path (71.4% WR) pro autobet.

#### B2: Zpřísnit team name matching
Přidat minimální délku 4 znaky pro substring match.
Přidat "whole word boundary" check — `"navi"` nesplní substring pokud target je `"navegadores"`.

#### B3: Přidat outcome freshness re-check
Před finálním bet placement, re-resolve leading team z AKTUÁLNÍHO score, ne ze score v momentě edge detekce.

### 🔮 PŘÍŠTÍ TÝDEN

## 6. MATEMATICKÝ ZÁVĚR

$$\text{Expected Daily P\&L} = \sum_i (\text{stake}_i \times (\text{WR}_i \times \text{odds}_i - 1))$$

**Současný stav:**
$$\bar{P\&L_{den}} = \frac{-36.93}{11} = -3.36 \text{ USD/den}$$

**Po fixech A1+A2+A3 (konzervativní odhad):**
- Eliminujeme esports edge + anomaly: saving ~$2.36/den
- CS2 explicit path odemčený: ~+$0.38/den (skálujem z 5W/2L za $7 volume)
- Tennis anomaly větší stakes: ~+$0.30/den (zdvojnásobíme volume)

$$\bar{P\&L_{den}^{fixed}} \approx -3.36 + 2.36 + 0.38 + 0.30 = -0.32 \text{ USD/den}$$

**To stále NENÍ PROFITABILNÍ.** Na profit potřebujeme:
1. CS2 explicit path s vyšším volume (fix B1)
2. Nové ověřené modely (Dota, LoL po validaci)
3. Větší bankroll + stakes na ověřených edgech

**Cesta k profitu: Nefixovat špatné modely — KONCENTROVAT se na model který funguje (CS2 edge 71.4% WR) a dát mu maximum traffic.**
